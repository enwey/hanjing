# 管理后台端业务功能未实现清单

检查日期：2026-06-26  
检查范围：`hanjing-clinic-管理后台源码-20260607` 管理后台前端、`hanjing-clinic-backend/routes/admin.js` 管理端接口、项目需求文档 `鼾静健康诊所_UI设计资料/06_小程序开发文档.md`。  
检查口径：不只看接口是否存在，重点按“后台业务是否能闭环、状态是否真实持久、操作是否能驱动小程序端和运营流程”判断。

## 一、总体结论

当前后台端已经具备较完整的管理后台页面结构，也有不少真实接口能力：登录、数据看板、预约列表、接诊工作台、病历录入、治疗建档、排班、门店、订单、提现、系统设置等模块都有部分落地。

但从诊所真实运营闭环看，仍有多处功能停留在“页面可看、按钮可点、局部接口可调”的阶段。主要缺口集中在：

- 收银订单没有真实支付/退款/库存/发票/资金流水闭环。
- 预约改约、新建预约、排班选择仍有硬编码时间和门店医生映射，未完全使用真实排班号源。
- 接诊病历、治疗建档、调整记录、随访任务分散提交，缺少一个事务化的“结束就诊”业务接口。
- 分销总览、推广员管理、推广商品管理仍大量使用本地假数据；提现审核前后端字段不匹配。
- 内容管理、文章编辑、轮播图管理仍是本地状态，没有真实发布、审核和上下架闭环。
- 权限、管理员、操作日志页面仍是本地数据，虽然后端有部分接口，但前端没有接入成可用的 RBAC 管理。
- 通知设置、到货通知、预约提醒、随访提醒等多数只是配置或 `console.log`，没有真实消息投递。
- 医疗数据导出、手机号解密、病历查看、订单导出等敏感操作缺少完整审计和审批。

## 二、P0：影响门诊核心经营闭环

### 1. 收银订单没有真实支付、退款、库存和资金流水闭环

**现状**

- 接诊工作台和预约列表都可以打开收银弹窗，并调用 `POST /api/admin/orders` 创建订单。
- 收银商品选项在前端硬编码，金额、折扣、商品明细由前端传入。
- 后端创建订单时直接写入 `pay_at = CURRENT_TIMESTAMP`，并按前端传入 `status` 标记为 `completed`、`processing` 或 `shipping`。
- 订单发货、到货通知、完成核销、退款审批都有接口，但多数只更新订单状态。
- 订单详情页的分销佣金、订单流程时间线、交易号等是静态展示。

**缺失的业务闭环**

- 真实收款：扫码/刷卡/微信/现金等支付方式没有对应支付流水、收款确认、退款通道。
- 商品校验：前端传什么商品和金额，后端就创建什么订单，缺少以后端商品、库存、价格为准的校验。
- 库存扣减：门店自提、快递发货、缺货预订没有库存占用、出库、入库、到货登记。
- 退款售后：退款接口只把状态改为 `refunded` 或 `paid`，没有退款申请、审核原因、微信退款、资金流水、库存回滚。
- 票据凭证：打印小票只是 Toast，没有真实小票、发票、收据、交易凭证。

**影响**

- 后台看起来能“收银成功”，但财务无法对账。
- 订单金额和佣金可能被前端篡改。
- 退款、发货、核销后没有完整的资金和库存追踪。

**建议优先级**

- 新增统一收银接口，由后端按商品 ID、数量、优惠规则计算金额。
- 建立 `payment_transactions`、`refund_records`、`inventory_movements` 等业务表或等价机制。
- 订单状态流转增加合法状态校验，禁止随意从任意状态改到完成/退款。
- 退款必须进入申请、审核、支付渠道退款、退款完成/失败流程。
- 小票/收据/发票生成后应有可追溯编号和打印记录。

### 2. 预约新建与改约没有完全使用真实号源

**现状**

- 预约列表已接入 `/api/admin/appointments`。
- 新建预约页面仍使用硬编码门店、医生、日期、时段，例如 `5月29日`、`09:30`。
- 新建预约时通过字符串判断映射 `storeId`、`doctorId`、`dateStr`，而不是从真实门店/医生/排班接口选择。
- 改约分支只 `PUT /api/admin/appointments/:id` 更新 `status: pending`，没有真正更新日期、时段、医生、排班。
- 预约列表 “今日” 统计和筛选有固定日期 `2026-05-29`。

**缺失的业务闭环**

- 选择门店 -> 选择医生 -> 读取真实排班 -> 选择可用号源 -> 创建预约并锁定号源。
- 改约 -> 释放旧号源 -> 锁定新号源 -> 记录改约日志 -> 通知患者。
- 超时未支付/未确认 -> 自动释放号源 -> 通知用户。

**影响**

- 管理员手动预约可能与真实排班冲突。
- 改约操作可能只改变状态，不改变实际就诊时间。
- 当天预约统计、门店筛选、医生筛选容易失真。

**建议优先级**

- 新建预约页改为读取 `/api/admin/stores`、`/api/admin/doctors`、`/api/admin/schedules`。
- 改约接口应接收并校验新 `schedule_id`、日期和时间。
- 后端创建/改约必须做并发号源校验和事务锁定。
- 预约筛选不要使用固定日期，统一使用当前日期或后端查询参数。

### 3. 接诊结束不是事务化业务接口

**现状**

- `appointment/emr.vue` 保存接诊时依次调用多个接口：新增病历、治疗建档或调整记录、创建随访任务。
- 后端 `POST /patients/:id/medical-records` 本身会写病历、更新患者病史、标记预约完成，并在处方包含 `HJ-MAD` 时自动创建治疗记录。
- 前端在病历成功后又可能额外调用治疗建档接口，存在重复或并发逻辑风险。
- 任何后续步骤失败，都可能出现病历已保存、治疗或随访未保存的半完成状态。

**缺失的业务闭环**

- 预检 -> 接诊 -> 病历 -> 治疗方案 -> 设备参数 -> 随访计划 -> 预约状态完成 -> 通知患者，应由一个“结束就诊”事务统一完成。
- 结束就诊后应触发小程序端病历、治疗记录、医嘱、复诊提醒同步。

**影响**

- 医生以为“保存并结束就诊”完成了全流程，但数据库可能只完成了其中一部分。
- 治疗档案可能重复创建，或随访任务漏建。
- 小程序端治疗页和病历页展示不稳定。

**建议优先级**

- 新增 `POST /api/admin/appointments/:id/complete-consultation` 聚合接口。
- 接口入参包含病历、治疗、调整、随访、通知，后端事务提交。
- 前端只调用一个结束就诊接口，失败时整体回滚并给出明确错误。
- 后端避免“处方自动建档”和前端“同步治疗建档”重复触发。

### 4. 提现审核前后端协议不一致

**现状**

- 提现审核页调用 `PUT /api/admin/distribution/withdraws/:id/status`。
- 前端传参是 `{ status: 'approved', remark: '...' }` 或 `{ status: 'rejected', remark: '...' }`。
- 后端读取的是 `const { approve } = req.body`，并根据布尔值决定通过或驳回。
- 由于前端没有传 `approve`，后端会把 `approve` 当作 falsy，导致“审核通过”可能被处理为失败/驳回。
- 小程序端申请提现时已经先扣减 `available_commission`，后台通过时后端又扣减一次，存在重复扣减风险。

**缺失的业务闭环**

- 提现申请 -> 冻结余额 -> 后台审核 -> 打款 -> 成功扣减冻结并增加已提现 -> 失败退回冻结余额。
- 当前余额、冻结中、已提现三类金额没有清晰分账。
- 没有真实微信/银行代付，也没有打款单号、失败原因和财务凭证。

**影响**

- 审核通过可能变成驳回。
- 推广员余额可能被重复扣减。
- 财务无法确认钱是否真实打出。

**建议优先级**

- 统一前后端字段：使用 `status` 或 `approve` 二选一，并补充自动化测试。
- 申请提现时只转入冻结金额，不直接从可提现永久扣除。
- 审核通过后记录打款流水，失败后退回可提现余额。
- 接入真实微信零钱/银行卡打款或至少记录线下付款凭证。

## 三、P1：影响运营、增长和服务质量

### 5. 分销总览、推广员管理、推广商品管理未真实落地

**现状**

- `distribution/index.vue` 的 KPI、排行榜、近期佣金记录全部是本地静态数据。
- `distribution/promoter.vue` 推广员列表和佣金明细由 `initialDistributors` 和 `Array.from` 生成。
- `distribution/products.vue` 推广商品、佣金比例、上下架都在前端数组里修改。
- 后端已有 `/api/admin/distribution/promoters`、`/leaderboard` 等部分接口，但前端没有完整接入。
- 后端创建订单时只按固定 10% 生成一级佣金，并立即 `settled`，没有读取系统设置中的佣金比例、二级佣金、结算周期。

**缺失的业务闭环**

- 推广员列表 -> 审核/禁用/等级调整 -> 推广关系 -> 订单归因 -> 一级/二级佣金 -> 冻结/结算/撤销。
- 推广商品 -> 设置佣金比例 -> 小程序推广商品同步 -> 下单按商品佣金生成佣金。

**影响**

- 后台无法真实运营分销体系。
- 商品佣金设置不会影响小程序端或订单佣金。
- 佣金立即结算会放大退款和售后风险。

**建议优先级**

- 分销总览和推广员管理接入真实接口。
- 推广商品管理接入 `products` 表中的 `is_distribution`、`commission_rate`、上下架状态。
- 佣金计算读取商品佣金和系统分销配置，支持一级/二级。
- 订单完成并过结算期后再从冻结变为可提现。

### 6. 内容管理和文章编辑仍是本地状态

**现状**

- `content/index.vue` 的文章/帖子列表由 `basePosts` 和 `Array.from` 生成。
- 审核通过、拒绝、置顶、删除只改本地数组。
- `content/edit.vue` 的保存草稿、发布、预览、换图都只是 Toast。
- 后端只有部分内容 GET 接口，文章创建/编辑/审核/上下架没有形成后台可用闭环。
- 富文本编辑器是 `contenteditable` 展示，工具栏按钮没有真实编辑命令或上传能力。

**缺失的业务闭环**

- 创建文章 -> 保存草稿 -> 上传封面/正文图片 -> 预览 -> 发布 -> 小程序展示。
- 用户社区内容 -> 待审核 -> 通过/驳回/下架 -> 小程序同步。
- Banner -> 上传图片 -> 设置跳转 -> 排序 -> 有效期 -> 上下架。

**影响**

- 后台内容管理不能驱动小程序端内容。
- 医疗健康内容审核不可追溯，合规风险高。
- 轮播图和文章运营无法真实发布。

**建议优先级**

- 补齐文章、社区帖子、评论、Banner 的增删改查和审核接口。
- 接入对象存储上传，保存封面图、正文图片和 Banner 图。
- 增加内容审核状态、审核意见、操作日志。
- 小程序端只展示已发布/已审核内容。

### 7. 随访任务独立页面未接真实接口

**现状**

- 患者详情页可以读取 `/patients/:id/follow-ups`。
- `patient/followup.vue` 独立随访页面仍使用本地 `tasks` 数组。
- 执行、改期、取消、新建随访都只是修改前端状态或 Toast。
- 后端已有随访任务和随访记录接口，但前端没有在该页面使用。

**缺失的业务闭环**

- 创建随访任务 -> 到期提醒 -> 执行随访 -> 记录沟通内容 -> 完成任务 -> 小程序/短信/微信触达患者。
- 改期和取消应有原因和审计记录。

**影响**

- 医生或客服在随访页面做的操作刷新后丢失。
- 随访任务无法驱动复诊和持续治疗。

**建议优先级**

- `patient/followup.vue` 接入真实随访接口。
- 执行随访时调用 `/follow-ups/records` 写沟通记录。
- 增加改期、取消接口和原因字段。
- 与通知系统联动，提醒患者复诊或佩戴打卡。

### 8. 排班有保存能力，但复制上月和真实时间规则未完成

**现状**

- 排班页可以读取和批量保存排班。
- “复制上月排班”只是成功提示，没有真实复制。
- 排班保存固定上午 `09:00-12:00`、下午 `14:00-18:00`，未使用系统设置中的营业时间/时段间隔。
- 批量保存整月时会发送所有日期和半天时段，风险较大。

**缺失的业务闭环**

- 门店营业时间 -> 医生排班 -> 时段生成 -> 预约号源 -> 调整排班时保护已有预约。
- 复制上月应读取上月排班并生成本月排班，同时跳过已有预约冲突。

**影响**

- 管理员以为复制上月成功，实际没有落库。
- 系统设置的时段间隔和营业时间无法真正影响排班。
- 保存整月排班时误操作成本较高。

**建议优先级**

- 实现真实复制上月接口和前端调用。
- 排班时段由后端根据门店营业时间、医生设置和系统时段间隔生成。
- 保存前展示变更摘要，特别是会清空/删除的排班。

### 9. 通知与消息投递没有真实闭环

**现状**

- 系统设置里有预约、提现、就诊、复诊通知开关。
- 订单到货通知接口只是 `console.log` 模拟微信订阅消息。
- 预约、改约、取消、接诊完成、随访、提现审核等操作没有统一通知事件。
- 数据库有 `user_notifications` 表，但管理端没有可配置模板、发送记录和失败重试。

**缺失的业务闭环**

- 后台业务事件 -> 站内通知 -> 小程序通知中心 -> 微信订阅消息/短信 -> 发送状态追踪。
- 通知开关、模板、提醒时间应真正控制发送行为。

**影响**

- 患者无法及时收到预约提醒、到货提醒、复诊提醒和提现结果。
- 管理员无法查看通知是否发送成功。

**建议优先级**

- 建立统一通知服务接口和发送记录。
- 管理端提供模板配置、开关、发送日志。
- 业务操作后写入通知事件，并异步投递微信订阅消息/短信。

## 四、P2：影响安全合规、统计准确性和维护

### 10. 权限、管理员、操作日志前端仍未接真实业务

**现状**

- 后端已有 `/api/admin/users`、`/api/admin/roles`。
- `settings/admin.vue` 管理员账号仍是本地 `baseAdmins`。
- `settings/permission.vue` 角色仍是本地 `baseRoles`。
- `settings/log.vue` 操作日志仍是本地数组，没有接后端审计日志。
- 菜单权限只在前端按 `role_code` 做简单过滤，按钮和接口级数据范围仍不完整。

**缺失的业务闭环**

- 创建管理员 -> 绑定角色和门店 -> 登录校验 -> 菜单/按钮/数据权限 -> 禁用后 token 失效。
- 敏感操作 -> 记录操作日志 -> 可筛选、导出、审计。

**影响**

- 后台权限页面不能真正管理系统权限。
- 敏感操作缺少可追溯审计。
- 不同岗位可能看到不该看的患者、订单、分销、财务数据。

**建议优先级**

- 管理员和角色页面接入真实接口。
- 后端补齐角色创建、编辑、禁用、权限资源维护接口。
- 前端菜单、路由和按钮从后端权限动态渲染。
- 操作日志页面接入真实 `admin_operation_logs` 或等价接口。

### 11. 患者列表和详情仍混有展示型推算数据

**现状**

- 患者列表接入真实患者接口，但最近就诊、来源、标签、病史等字段仍有不少本地推断或固定值。
- 患者详情中诊断、治疗方案、推荐人、下一次随访、科室分布等存在默认值或推算值。
- 睡眠趋势图使用佩戴舒适度推算 AHI：`30 - comfort * 4`，不是检测数据。

**缺失的业务闭环**

- 患者档案 -> 预约 -> 病历 -> 评估 -> 治疗 -> 佩戴数据 -> 随访 -> 复诊，应统一形成真实患者画像。
- 标签、来源、推荐人、复诊状态应来自后端业务记录。

**影响**

- 运营人员可能基于推算或默认信息做错误判断。
- 医疗疗效趋势不可信，不能作为医生判断依据。

**建议优先级**

- 后端患者详情聚合真实病历、治疗、评估、订单、随访和分销来源。
- 去掉前端推算 AHI 等医疗指标，改用真实评估或医生录入数据。
- 标签、来源、推荐人、最近就诊由后端返回。

### 12. 导出功能多数只是前端本地导出或 Toast

**现状**

- 预约列表有前端 CSV 导出。
- 订单、分销、患者、门店报表等多个导出按钮只是成功提示。
- 导出没有服务端权限校验、字段脱敏、审批、审计和水印。

**缺失的业务闭环**

- 导出申请 -> 权限校验 -> 脱敏策略 -> 生成文件 -> 下载 -> 审计日志。

**影响**

- 医疗健康和财务数据导出存在合规风险。
- 大数据量导出会卡前端，且无法追溯导出行为。

**建议优先级**

- 敏感数据导出统一走后端。
- 根据角色控制可导出字段。
- 所有导出写操作日志，必要时加审批。

### 13. 系统设置保存了配置，但未全部驱动业务

**现状**

- 系统设置可以读取和保存到 `system_settings`。
- 预约提前天数已被部分小程序端排班日期使用。
- 佣金比例、提现手续费、最低提现、自动审批、通知开关、预约时间段等配置未全面参与后端业务计算。

**缺失的业务闭环**

- 设置保存后应立即影响预约规则、分销佣金、提现校验、通知投递、排班生成。

**影响**

- 管理员以为设置已生效，但实际业务仍按硬编码或默认逻辑运行。

**建议优先级**

- 梳理每个设置项的消费点，补齐后端读取逻辑。
- 设置变更写操作日志。
- 高风险设置，例如佣金比例、提现自动审批、预约定金，应增加二次确认和权限限制。

## 五、建议实施顺序

1. 修复 P0 协议错误和资金风险：提现审核字段、余额冻结/扣减、收银金额后端校验、退款状态流转。
2. 重构预约和接诊闭环：真实号源新建/改约、事务化结束就诊、病历/治疗/随访统一提交。
3. 补齐订单经营闭环：支付流水、退款记录、库存、发货、核销、票据。
4. 落地分销真实运营：推广员、推广商品、佣金冻结/结算、二级佣金、结算周期。
5. 落地内容和通知：文章/Banner/社区审核、消息模板、发送记录、失败重试。
6. 接入权限和审计：管理员、角色、菜单/按钮/数据权限、操作日志、敏感导出。
7. 清理演示数据：移除前端静态 KPI、静态趋势、静态佣金、静态文章和本地随访任务。
