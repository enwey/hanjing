# 微信小程序端业务功能未实现清单

检查日期：2026-06-26  
检查范围：`mp-weixin` 小程序端、`hanjing-clinic-backend/routes/client.js` C 端接口、项目需求文档 `鼾静健康诊所_UI设计资料/06_小程序开发文档.md`。  
检查口径：不只看接口是否存在，重点按“用户业务是否能闭环、数据是否真实持久、状态是否能被后续业务消费”判断。

## 一、总体结论

当前小程序端已经具备较完整的页面壳、路由结构和部分真实 API 对接，预约、ESS 量表、鼾声录音、治疗打卡、社区、在线客服等模块也有一定业务实现。

但从生产业务闭环看，仍有多处功能停留在“演示可点”或“只完成局部状态”的阶段。最主要的缺口是：

- 商城购买与订单支付链路没有实现，商品页只提供咨询入口。
- 预约支付存在字段协议不匹配风险，且订阅消息、就诊提醒、爽约/完成后续流程未闭合。
- AI 鼾声分析虽然已能录音并提交文件内容，但分析结果仍是后端随机生成，不是真实 AI/医学分析。
- 治疗追踪只实现了佩戴打卡的局部数据，睡眠报告、调整记录、医嘱随访仍大量使用静态逻辑。
- 分销中心有展示、提现申请、团队/订单列表，但缺少关系绑定、邀请落地页、购买归因、佣金冻结/结算闭环。
- 通知中心前端调用了通知接口，但后端 C 端缺少对应通知接口，订阅消息也没有接入。
- 家庭成员、病历、设备、在线客服等模块有页面和部分数据，但缺少医疗业务所需的建档、授权、审核、追踪和服务状态。

## 二、P0：影响核心交易与医疗服务闭环

### 1. 商城购买、订单创建、支付、物流与售后未实现

**现状**

- `pages/product/index.js` 商品列表是页面内硬编码数据，没有使用 `api.getProducts()`。
- `pages/product/detail/index.js` 商品详情同样是硬编码商品字典。
- 商品详情“加入咨询”只把本地按钮置为“已加入”，另一个操作只提示“请前往门诊或联系顾问下单”。
- `pages/order/index.js` 和 `pages/order/detail.js` 只展示订单，没有创建订单、继续支付、取消订单、确认收货、退款/售后等操作。
- C 端 API 层只有 `getOrders/getOrderDetail/getProducts/getProductDetail`，没有小程序端创建订单、发起商品支付、确认支付、收货地址、售后申请等能力。

**缺失的业务闭环**

- 商品浏览 -> 选择规格/数量 -> 收货地址 -> 创建订单 -> 微信支付 -> 支付回调 -> 发货 -> 确认收货 -> 完成订单。
- 服务套餐类商品购买后也没有绑定预约、治疗计划或权益。
- 订单完成后没有触发分销佣金冻结/结算，也没有触发会员成长值、积分或通知。

**影响**

- “健康商城”无法线上成交，只能作为展示页。
- 订单页只能看已有数据，用户无法完成实际交易。
- 分销系统没有真实订单来源，佣金链路无法成立。

**建议优先级**

- 先补齐 C 端 `POST /orders`、`POST /orders/:id/pay`、支付回调、订单状态变更接口。
- 小程序商品详情改为真实 API 数据，增加收货地址、创建订单、支付、订单操作按钮。
- 医疗器械与服务套餐要区分交付规则：实物走物流，服务套餐走预约/核销/权益。

### 2. 预约支付和预约后续业务未闭合

**现状**

- 预约选择门店、医生、日期、时段、确认页面已具备基本流程。
- `pages/appointment/confirm.js` 在创建预约后判断 `t.requireDeposit`，但后端 `POST /appointments` 返回的预约对象未体现该字段，实际是否支付依赖前后端字段是否一致。
- `api.payAppointmentDeposit()` mock 返回 `{ success: true }`，但真实 `wx.requestPayment` 需要 `timeStamp/nonceStr/package/signType/paySign`。
- `pages/appointment/index.js` 和 `pages/appointment/detail.js` 有支付按钮逻辑，但支付参数、支付后状态同步和后端订单/预约账务仍需确认。
- `getBookingSettings()` 可读取预约定金设置，但前端确认页创建预约后没有用本地设置决定支付，而是判断创建预约返回值。

**缺失的业务闭环**

- 预约创建 -> 根据系统设置生成待支付/已确认状态 -> 微信支付 -> 支付确认 -> 预约确认 -> 订阅消息提醒。
- 改约/取消后没有完整处理退款、订阅消息、排班释放、取消原因统计。
- 就诊完成后应由后台生成治疗记录、病历、时间线，小程序端未展示与预约完成状态的联动入口。

**影响**

- 如果后端开启定金，前端可能不会进入支付；如果进入支付，支付参数不完整会导致支付失败。
- 预约支付、挂号费、定金、退款和预约状态可能不一致。
- 用户完成预约后收不到标准提醒，运营侧难以降低爽约率。

**建议优先级**

- 统一预约创建返回字段：`status`、`requireDeposit`、`depositAmount`、`payOrderId`、`payParams`。
- 前端以真实返回决定是否支付，不要混用本地设置和预约对象字段。
- 接入 `wx.requestSubscribeMessage`，覆盖预约成功、改约、取消、就诊前提醒。
- 后台完成就诊后，自动生成治疗记录、病历、医嘱和通知。

### 3. AI 鼾声分析不是医学/AI真实分析

**现状**

- `pages/assessment/recording/index.js` 已接入 `wx.getRecorderManager()`，能录音、停止并读取本地 `tempFilePath`。
- 提交时把音频读成 base64 后通过 JSON 传给 `/assessments/snore`。
- 后端 `POST /api/v1/assessments/snore` 会保存文件，但平均分贝、峰值、鼾声占比、呼吸暂停事件和风险等级仍是随机计算。
- 前端没有录音最短时长、夜间录制建议、试听、重录、上传进度、后台分析排队/失败重试等体验。

**缺失的业务闭环**

- 真实音频采集 -> 文件上传/对象存储 -> AI 分析任务 -> 风险结果 -> 报告解释 -> 推荐预约/复诊 -> 后续医生查看。
- 当前分析结果不可作为医疗健康建议依据，也没有医生复核机制。

**影响**

- “AI鼾声分析”核心卖点仍是演示算法，风险结果可能误导用户。
- base64 JSON 上传对大文件不友好，真机弱网下容易失败。
- 医生端无法基于可靠结果做诊前判断。

**建议优先级**

- 改为 `wx.uploadFile` 或直传对象存储，并返回文件 URL/任务 ID。
- 后端引入真实音频分析服务或至少固定算法，不要随机生成医疗风险指标。
- 增加录音有效性校验：时长、音量、格式、背景噪音、是否过短。
- 报告页标注“筛查参考”，并提供预约/复诊/咨询转化入口。

## 三、P1：影响长期治疗、复购与运营

### 4. 治疗追踪只有打卡局部真实，报告和调整记录仍未业务化

**现状**

- `pages/treatment/index.js` 可读取治疗记录、佩戴记录、汇总和时间线，并能提交当日佩戴打卡。
- `pages/treatment/sleep-report/index.js` 使用固定的周/月指标、固定趋势数组和本地算法生成睡眠健康报告。
- `pages/treatment/adjust-detail/index.js` 调整历史是页面内静态数组，只取治疗记录里的当前设备型号和前移量。
- `doctor-advice`、`timeline`、`sleep-trend` 页面主要是展示，没有形成医生调整、患者反馈、复诊预约之间的业务联动。

**缺失的业务闭环**

- 医生创建治疗方案 -> 患者每日佩戴/反馈 -> 系统生成趋势 -> 医生查看反馈 -> 调整设备参数 -> 推送复诊提醒 -> 更新治疗阶段。
- 设备调整记录、维护记录、医嘱、随访和睡眠报告之间没有统一数据模型。

**影响**

- 用户能打卡，但无法获得可信的阶段性治疗反馈。
- 医生端的治疗动作难以在小程序端完整体现。
- 复诊提醒和设备调整无法驱动预约转化。

**建议优先级**

- 增加治疗报告接口，基于真实佩戴记录、舒适度、鼾声/评估数据生成。
- 增加设备调整记录接口，替换静态数组。
- 将低舒适度、低依从率自动转化为复诊/咨询入口。
- 医嘱、反馈、维护记录加入医生回复与处理状态。

### 5. 分销没有完成“邀请-绑定-下单-佣金-提现”闭环

**现状**

- 分销中心、团队、佣金、提现、推广商品、规则页面存在。
- `pages/distribution/invite/index.js` 可复制邀请码，但“保存海报”只是 Toast，没有生成带小程序码的推广海报。
- `pages/distribution/products/index.js` 只有推广商品列表，没有商品推广详情页和分享参数落地页。
- `applyWithdraw()` 只提交提现金额，缺少账户绑定、实名校验、审核进度、失败原因等流程。
- 小程序端没有处理 `scene` 或邀请码参数，也没有在用户注册、预约、下单时绑定上下级关系。

**缺失的业务闭环**

- 推广员生成邀请链接/海报 -> 新用户扫码进入 -> 绑定关系 -> 下单/预约 -> 佣金冻结 -> 订单完成 N 天后结算 -> 可提现 -> 财务审核 -> 打款/失败。
- 一级/二级佣金、结算周期、退单撤佣没有在前端业务里体现。

**影响**

- 分销看起来有页面，但无法真实带来增长和结算。
- 推广员无法有效分享，运营无法追踪来源。
- 佣金与订单状态脱节，财务风险高。

**建议优先级**

- 小程序 `onLaunch/onShow` 解析推广参数并调用关系绑定接口。
- 邀请页生成真实小程序码海报并支持保存相册。
- 商品/预约下单携带推广来源，订单完成后生成佣金冻结记录。
- 提现页补实名、账号、最低提现、手续费、审核状态和失败原因。

### 6. 通知中心和订阅消息未接通

**现状**

- `api/index.js` 定义了 `getNotifications`、`markAllNotificationsRead`、`markNotificationRead`。
- `pages/profile/notifications/index.js` 调用这些方法。
- 后端 C 端路由清单中未看到 `/api/v1/user/notifications`、`/read-all`、`/:id/read`。
- 项目文档要求预约、治疗、随访、佣金、提现等通知，但小程序端没有 `wx.requestSubscribeMessage`。

**缺失的业务闭环**

- 业务事件 -> 站内通知 -> 未读数 -> 用户点击跳转对应业务详情。
- 业务事件 -> 微信订阅消息 -> 用户授权 -> 模板消息发送。

**影响**

- 通知页上线会接口失败。
- 预约提醒、复诊提醒、提现审核等关键运营触达无法实现。
- 用户错过就诊或复诊时，系统没有主动唤回能力。

**建议优先级**

- 后端补齐 C 端通知接口。
- 建立业务事件通知表：类型、标题、内容、跳转路径、是否已读。
- 在预约成功、取消、改约、就诊完成、复诊提醒、佣金入账、提现审核后写入通知。
- 接入订阅消息授权与发送。

## 四、P2：影响资料完整性、服务质量与合规

### 7. 家庭成员和患者建档信息不足

**现状**

- 家庭成员支持新增、列表、删除。
- 新增只校验姓名，手机号、年龄、关系、性别没有严格校验。
- 不能编辑成员信息，也没有就诊人选择、默认就诊人、成员授权关系。
- 预约确认页自动取第一个 `self` 成员或列表第一个，用户不能主动选择就诊人。

**缺失的业务闭环**

- 建档 -> 选择就诊人 -> 预约/评估/病历归属 -> 家属授权查看与管理。
- 儿童/老人等家属就诊场景没有授权和隐私边界。

**影响**

- 多就诊人家庭容易出现病历、预约、评估结果归属错误。
- 医疗信息合规风险较高。

**建议优先级**

- 预约、评估、治疗入口都支持选择就诊人。
- 增加成员编辑、默认就诊人、手机号/年龄/关系校验。
- 对非本人档案增加授权声明和可见范围。

### 8. 病历档案只读展示，未与预约/治疗完整关联

**现状**

- `pages/profile/medical-records/index.js` 只读取病历列表并展示诊断、处方、备注。
- 用户无法查看完整病历详情、检查附件、医生签名、处方附件。
- 预约完成后生成病历、治疗记录、医嘱之间的关联主要依赖后端/后台，前端缺少业务入口。

**缺失的业务闭环**

- 预约完成 -> 生成病历 -> 生成治疗记录 -> 生成医嘱/复诊计划 -> 小程序可查看并触发复诊。

**影响**

- 医疗服务完成后用户获得感不足。
- 后续复诊和治疗追踪缺少可信依据。

**建议优先级**

- 增加病历详情页。
- 病历关联预约、医生、治疗记录、设备调整、附件。
- 对敏感信息做脱敏、水印或二次确认。

### 9. 在线客服是轮询消息，缺少服务状态和消息可靠性

**现状**

- `pages/profile/online-service/index.js` 每 4 秒轮询 `/im/messages`。
- 发送消息调用 `/im/send`。
- 前端没有消息发送中/失败/重发状态，也没有客服在线状态、分配医生/客服、未读数。

**缺失的业务闭环**

- 用户发起咨询 -> 分配客服/医生 -> 消息可靠送达 -> 已读/未读 -> 结束会话 -> 关联预约或售后工单。

**影响**

- 弱网或接口失败时，用户以为消息已发送但实际可能丢失。
- 客服运营无法区分待回复、已处理、超时会话。

**建议优先级**

- 消息增加本地临时 ID、发送中、失败、重试状态。
- 后端返回会话 ID，并支持分页拉取历史消息。
- 增加在线状态、未读数、工单/预约转化入口。

### 10. 社区内容审核和风控仍不足

**现状**

- 社区发帖、评论、点赞已接入真实接口。
- 后端当前发帖和评论直接 `approved`。
- 小程序端没有举报、删除、屏蔽、医生身份认证说明。

**缺失的业务闭环**

- 用户发布 -> 内容审核 -> 展示/驳回 -> 举报 -> 管理端处理。
- 医疗健康内容需要避免违规疗效承诺、广告、联系方式导流。

**影响**

- 医疗健康社区存在审核和合规风险。
- 普通用户与专家/医生身份边界不清。

**建议优先级**

- 接入内容安全审核，默认待审核或先机审后公开。
- 增加举报、删除、屏蔽和审核状态提示。
- 医生/专家身份由后端返回，不允许前端自行决定。

## 五、横向问题

### 1. API 层仍混入 Mock 回调和生产逻辑

**现状**

- `api/index.js` 每个方法都保留 mockCallback。
- `api/request.js` 当前 `ENABLE_MOCK_FALLBACK = false`，生产不会 fallback，但 mock 代码仍随包存在。
- 前端页面中仍有不少硬编码数据，与真实 API 数据并存。

**影响**

- 开发、演示、生产逻辑混在一起，后续容易出现“页面看似可用但没有真实数据”的误判。
- 包体积和维护成本增加。

**建议**

- 将 mock 模式抽离为独立 demo 配置或独立 mock service。
- 生产构建不打包 mock 数据。
- 页面数据统一从 API 或 store 获取，避免同一模块一半真实一半静态。

### 2. 医疗健康数据的校验、脱敏和授权边界不足

**现状**

- 用户、家庭成员、病历、评估、治疗、录音等都涉及敏感信息。
- 小程序端缺少统一表单校验、隐私提示、查看敏感资料的二次确认。
- 本地仍直接使用 `access_token` 存储。

**影响**

- 医疗健康类应用上线和运营存在合规风险。
- 家属档案、录音、病历等信息容易越权或误归属。

**建议**

- 建立统一的表单校验和敏感字段处理规则。
- 病历、录音、身份证/手机号等敏感信息按需脱敏。
- 增加就诊人授权、家庭成员管理授权、隐私协议正式页面。

## 六、建议拆分实施顺序

1. 先完成交易和预约支付：商品真实数据、订单创建、微信支付、预约支付字段统一。
2. 补齐通知体系：站内通知接口、订阅消息授权、预约/治疗/佣金事件触发。
3. 完成治疗追踪业务化：真实报告、调整记录、医嘱反馈、复诊提醒。
4. 完成分销闭环：邀请参数、关系绑定、归因下单、佣金结算、提现审核。
5. 强化医疗合规：正式协议、隐私授权、家庭成员授权、病历脱敏、内容审核。
6. 清理演示代码：移除生产包 mock、硬编码商品/报告/调整记录，统一 API 数据源。
