# 鼾静健康诊所 - 系统版本与上线记录

本文件归档了鼾静健康诊所微信小程序与后台管理服务的开发迭代日志、用户需求变更记录，以及正式上线前的生产环境风险排查清单。

---

## 1. 迭代日志 (Iterative Changelog)

以下是系统近期核心开发周期的版本记录：

### v1.1.0-RC1 (当前版本 - 2026-07-02)
* **安全加固 (Security Hardening)**：
  * 全线接入 PII 敏感健康数据 AES-256-CBC 确定性对称加密，对 `phone` 与 `id_card` 密文落库，防范物理拖库风险；
  * 对静态托管上传目录 `/uploads` 部署 Nginx 与 Node.js 结合的 Referer 防盗链与 JWT 签名双重身份拦截器；
  * 对小程序本地缓存 `access_token` 进行运行时异或混淆，防备越狱设备数据被篡改或窃取。
* **稳定性重构 (Stability Patches)**：
  * 限制文件上传后缀，防范通过穿越路径（如 `?ext=../../server.js`）覆盖主服务器脚本的 Path Traversal 漏洞；
  * 就诊挂号支付同步接口中追加事务行级排他锁，杜绝弱网双击引起的并发逻辑重复执行。
* **业务逻辑优化 (Business Logic)**：
  * 静默分销激活：普通新用户首次点进分销海报时自动激活，提供唯一邀请码；
  * 完成实名认证时，才可激活并领取 2000 体验积分包；
  * 新增未成年挂号阻鼾器时的前后台双重科普引导拦截。

## 2. 需求变更记录 (Requirement Changes)

记录系统在迭代过程中，用户对产品策略和具体页面作出的核心改动：

1. **废除所有默认兜底数据 (No Default Mocking)**：
   * *原设定*：商品卡片无图、分销列表无图、视频直播间未接入真实推流时，前端小程序采用静态默认占位图（`/static/product/hj-mad-03.png`）和本地视频兜底。
   * *变更后*：严禁任何虚假兜底，未配置图则展示为空，拉流未开始时小程序端进行对应“暂无推流”友好状态交互，完全反映原生后端接口数据结构。
2. **积分获取门槛由注册赠送变更为实名专享**：
   * *原设定*：新用户小程序注册成功即送 2000 积分。
   * *变更后*：新用户注册后分数为 0，必须提供真实姓名与 18 位身份证完成实名核验（绑定 user_id 和 `relation='self'`）才赠送一次体验积分包，强化财务安全性。

## 3. 上线风险清单 (Launch Risk Checklist)

在正式割接投产前，运维团队必须逐一核对以下 5 大系统上线安全风险，并勾选确认：

- [ ] **[风险1] 微信 API 合法域名配置缺失**
  * *描述*：小程序内只能请求已在公众平台登记的 `https` 域名。
  * *防范*：在微信公众平台后台将生产 API 服务域名（如 `https://api.snore-clinic.com`）配置到 `request合法域名` 和 `uploadFile合法域名` 中，否则上线后小程序将发生接口断连。
- [ ] **[风险2] 商户支付私钥明文泄漏**
  * *描述*：微信支付 APIv3 签名所需的 `apiclient_key.pem` 如果直接托管在 Git 公共仓库，会导致密钥泄露和资金拦截风险。
  * *防范*：此私钥文件严禁入库。应通过环境变量或仅在服务器本地生产部署时写入私有配置目录中。
- [ ] **[风险3] 生产环境数据库 PII 加密密钥重置**
  * *描述*：若采用默认的 `hanjing_clinic_pii_secret_key_32` 秘钥上线，由于该秘钥已开源，攻击者可对被盗数据库直接解密。
  * *防范*：上线前，必须在服务器环境的 `.env` 中，将 `PII_ENCRYPTION_KEY` 变量修改为专属的 32 位高强度随机哈希字符串，并严格封存。
- [ ] **[风险4] 磁盘空间爆满导致服务锁死**
  * *描述*：鼾声录音和就诊病历附件持续上传，加上高吞吐访问日志，极易撑爆云主机系统盘。
  * *防范*：
    1. 日志强制应用 `pm2-logrotate` 进行每日自动切分和保留 30 天；
    2. 将静态托管的本地 `uploads/` 迁移至云端私有对象存储（OSS），实现无盘化物理部署。
- [ ] **[风险5] 定时任务断网时锁死状态**
  * *描述*：结算佣金的后台定时器在服务器重启或 MySQL 临时中断时若无容错，会引发程序死锁。
  * *防范*：确保 Nodemon 或 PM2 守护配置生效，并已通过后台的全局 `try/catch` 告警链路对接钉钉/飞书。
