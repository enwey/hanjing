# 页面改造对齐规范

适用范围：`mp-weixin-rewrite`

对标基准：`mp-weixin`

目标：后续所有页面修改，默认按照 `mp-weixin` 做一比一对齐，并用本清单做实现前确认与修改后验收。

## 一、总原则

1. `mp-weixin` 是唯一对标源，`mp-weixin-rewrite` 必须按一比一还原处理。
2. 一比一不只看功能，还要看结构、样式、文案、数据、交互是否一致。
3. 用户点名改哪个页面，就只改该页面及其直接必需依赖，不擅自扩展到其它页面。
4. 未被明确要求的优化不主动做，包括但不限于：重构、换数据源、改文案、统一其它页面样式。
5. 如果发现关联页面也有同类问题，统一放到“相关但未修改”中说明，不直接修改。

## 二、修改前检查清单

每次接到页面修改需求，先完成以下对照：

1. 找到 `mp-weixin` 中对应页面，确认真实参考文件。
2. 对比 `mp-weixin-rewrite` 当前实现，先判断差异属于哪一类：
   - 功能差异
   - 结构差异
   - 样式差异
   - 数据差异
   - 交互差异
3. 明确当前页面的数据来源：
   - 接口地址是否一致
   - 请求参数是否一致
   - 字段映射是否一致
   - 列表截取和排序是否一致
4. 明确当前修改的最小范围，只改满足需求所必需的文件。

## 三、功能对齐要求

1. 页面模块要和 `mp-weixin` 保持一致：
   - 模块是否存在
   - 模块顺序是否一致
   - 默认展示逻辑是否一致
2. 页面交互要和 `mp-weixin` 保持一致：
   - 点击跳转目标
   - 按钮行为
   - 列表项点击行为
   - 加载、刷新、重试逻辑
3. 状态展示要和 `mp-weixin` 保持一致：
   - 加载态
   - 空态
   - 错误态
   - 禁用态

## 四、数据对齐要求

1. 不能只把静态布局做出来，必须确认页面真实拿到数据并能渲染。
2. 列表数据要核对以下内容：
   - 是否显示
   - 显示几条
   - 排序方式
   - 截取规则
   - 字段内容
   - 字段格式化规则
3. 如果原版页面带有位置、身份、状态等上下文参与请求，重写页也必须保持一致。
4. 任一接口失败时，要确认是否影响整页展示，降级策略要尽量贴近原版。

## 五、视觉对齐要求

以下内容默认都要和 `mp-weixin` 对齐，而不是“看起来差不多”：

1. 文案内容
2. 字体
3. 字号
4. 字重
5. 行高
6. 字体颜色
7. 图标类型和大小
8. 按钮样式
9. 卡片结构
10. 圆角
11. 阴影
12. 边框
13. 模块间距
14. 内边距
15. 页面留白节奏

## 六、顶部区域专项检查

顶部区域是高频偏差点，每次都要重点核对：

1. 安全区处理是否正确
2. 标题位置是否和原版一致
3. 副标题或描述位置是否一致
4. 顶部 Banner 或头图区块高度是否一致
5. 顶部悬浮入口、快捷入口、吸顶区域是否一致
6. 顶部内容是否过于贴边或下沉

## 七、列表卡片专项检查

医生、门店、商品、服务等列表卡片，默认核对以下项：

1. 卡片整体结构是否一致
2. 封面图、头像、标题、副标题位置是否一致
3. 标签数量、样式、间距是否一致
4. 主信息和辅助信息展示顺序是否一致
5. 是否出现原版没有的字段
6. 是否遗漏原版已有的字段
7. 卡片点击区域和反馈效果是否一致

## 八、修改后验收清单

每次修改完成后，至少验证以下内容：

1. 页面可以正常编译
2. 页面无新的运行时报错
3. 接口可以正常请求
4. 页面真实显示数据
5. 核心模块完整显示
6. 样式与 `mp-weixin` 基本一致
7. 本次修改没有误伤相邻功能

## 九、交付说明格式

后续每次页面修改完成，默认按以下结构汇报：

1. 已修改内容
2. 对照 `mp-weixin` 的结论
3. 验证结果
4. 相关但未修改

## 十、执行约定

后续你只要直接说“改某个页面”，默认我会：

1. 先对比 `mp-weixin` 对应页面
2. 再修改 `mp-weixin-rewrite` 指定页面
3. 改完做功能验证和视觉验证
4. 最后按本规范输出结果
