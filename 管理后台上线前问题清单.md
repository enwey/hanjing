# 管理后台上线前问题清单

本文档基于当前 `hanjing-clinic-管理后台源码-20260607` 目录进行检查，目标是对齐上线前生产级后台系统要求。当前后台为 Vue 3 + TypeScript + Vite + TDesign，构建可以通过，但仍存在较多演示态、本地硬编码、权限和安全闭环不足的问题。

## 一、总体结论

当前管理后台已经具备基本工程结构和部分真实接口能力：

- 使用 Vue 3、Vue Router、Pinia、TDesign、Axios。
- 已有预约、患者、医生、门店、订单、诊疗录入等页面。
- 部分页面已接入 `/api/admin` 后端接口。
- `npm run build` 可成功构建。

但从上线前标准看，当前后台仍不建议直接部署生产，主要风险集中在：

- 鉴权仅校验本地 token 是否存在，没有完整权限控制。
- 短信登录、权限管理、管理员管理、退款审核、提现审核、内容审核等关键模块仍是前端硬编码或本地状态。
- 医疗健康敏感数据未形成脱敏、权限、审计和导出管控闭环。
- API 层缺少统一业务模块、请求取消、重复提交保护和生产日志策略。
- 构建产物存在大 chunk，首屏和运行性能需要优化。
- 缺少生产部署安全配置、CSP、反向代理、HTTPS、错误监控和操作审计闭环。

## 二、上线前必须整改的问题

### 1. 前端路由鉴权过于简单

**问题位置**

- `hanjing-clinic-管理后台源码-20260607/src/router/index.ts`

**当前表现**

路由守卫只判断：

```ts
const token = localStorage.getItem('auth_token')
if (to.name !== 'login' && !token) {
  next({ name: 'login' })
}
```

**为什么不符合上线要求**

只要浏览器本地存在任意 `auth_token` 字符串，就能进入后台页面。前端没有校验 token 是否真实有效，也没有获取当前用户、角色、门店范围和权限菜单。

**整改要求**

- 进入后台前调用 `/api/admin/me` 校验 token。
- token 无效、过期、用户禁用时清理登录态并跳转登录页。
- 根据后端返回的角色和权限生成可访问路由与菜单。
- 对详情页、二级页、操作按钮做权限判断。
- 后端仍必须做最终权限校验，前端权限只做展示与体验控制。

**建议验收标准**

- 手动写入假 token 不能进入后台。
- 禁用账号再次访问会被踢出。
- 不同角色只能看到和访问授权页面。
- 直接输入未授权 URL 会跳转 403 或无权限页。

### 2. 短信登录是 mock，存在绕过真实认证风险

**问题位置**

- `hanjing-clinic-管理后台源码-20260607/src/views/login/index.vue`

**当前表现**

- `sendSms()` 没有调用后端发送验证码接口，只启动倒计时并提示发送成功。
- 短信登录分支直接写入 `mock_jwt_token_for_hanjing_clinic`。

**为什么不符合上线要求**

这相当于任何人输入任意手机号和验证码都能模拟登录。生产环境必须彻底移除。

**整改要求**

- 接入真实短信验证码发送接口。
- 接入验证码登录接口。
- 后端校验验证码、过期时间、频率限制、IP 风控。
- 登录成功后返回真实 token 和用户信息。
- 开发/演示快捷登录必须通过环境开关隔离，不能进入生产构建。

**建议验收标准**

- 任意验证码不能登录。
- 验证码过期、错误、频繁发送都有明确限制。
- 生产包中不存在 mock token 登录逻辑。

### 3. 管理员账号管理仍是本地数据

**问题位置**

- `hanjing-clinic-管理后台源码-20260607/src/views/settings/admin.vue`

**当前表现**

- 管理员列表来自 `baseAdmins` 和 `Array.from` 生成。
- 新增、编辑、禁用只修改前端 `admins.value`。
- 不调用 `/api/admin/users`。

**为什么不符合上线要求**

管理员账号是后台安全核心模块，必须真实落库并具备审计、角色绑定、状态控制和密码/登录策略。

**整改要求**

- 接入管理员列表、创建、编辑、禁用、重置密码接口。
- 禁用账号后后端立即使 token 失效。
- 创建管理员时绑定角色和门店范围。
- 操作写入审计日志。
- 超级管理员保护逻辑放到后端。

**建议验收标准**

- 刷新页面后管理员变更仍存在。
- 被禁用账号无法继续访问接口。
- 管理员操作能在操作日志中追溯。

### 4. 权限管理仍是本地角色数据

**问题位置**

- `hanjing-clinic-管理后台源码-20260607/src/views/settings/permission.vue`
- `hanjing-clinic-管理后台源码-20260607/src/views/settings/role-edit.vue`

**当前表现**

- 角色列表由 `baseRoles` 和 `Array.from` 生成。
- 启用/禁用角色只修改本地状态。
- 菜单与路由没有按角色动态生成。
- 未看到按钮级权限和数据范围权限。

**为什么不符合上线要求**

诊所后台至少会涉及超级管理员、门店管理员、医生、客服、财务、内容运营等角色。没有 RBAC 会导致越权访问患者、订单、财务和病历数据。

**整改要求**

- 接入真实角色和权限接口。
- 建立菜单权限、按钮权限、数据权限。
- 后端按角色和门店范围校验接口。
- 前端菜单由权限配置动态渲染。
- 角色变更后用户权限实时生效或强制重新登录。

**建议验收标准**

- 财务角色不能访问病历详情。
- 内容角色不能访问订单和提现。
- 门店管理员只能看本门店数据。
- 未授权接口返回 403。

### 5. 退款审核仍是本地 mock 状态

**问题位置**

- `hanjing-clinic-管理后台源码-20260607/src/views/order/refund.vue`

**当前表现**

- 退款列表由前端数组生成。
- 审核通过/拒绝只修改本地 `row.status`。
- 没有调用真实退款审核接口。
- 没有对接支付退款、库存、订单状态、售后记录。

**为什么不符合上线要求**

退款涉及资金安全，不能停留在前端状态变化。

**整改要求**

- 接入退款申请列表接口。
- 审核通过需调用后端退款流程。
- 后端对接微信支付退款或线下退款流程。
- 拒绝退款需记录原因。
- 退款结果与订单状态、资金流水、操作日志同步。
- 金额不可由前端信任，必须以后端订单金额为准。

**建议验收标准**

- 审核后刷新状态仍正确。
- 支付渠道退款状态可追踪。
- 重复审核、重复退款被后端拦截。

### 6. 提现审核仍是本地 mock 状态

**问题位置**

- `hanjing-clinic-管理后台源码-20260607/src/views/distribution/withdraw.vue`

**当前表现**

- 页面明确标注“提现记录 mock 数据”。
- 提现记录由前端数组生成。
- 批量审核和驳回只修改本地状态。
- 未调用后端 `/api/admin/distribution/withdraws`。

**为什么不符合上线要求**

提现涉及推广佣金和资金结算，必须有真实审核、风控、财务凭证和审计。

**整改要求**

- 接入提现列表、详情、通过、驳回接口。
- 审核通过前校验账户、余额、佣金结算状态。
- 支持驳回原因。
- 支持批量操作但必须逐笔后端事务处理。
- 记录审核人、审核时间、IP、结果。
- 涉及银行卡号、微信提现账号必须脱敏展示。

**建议验收标准**

- 刷新后审核状态不丢失。
- 余额不足、重复审核、异常账号无法通过。
- 财务可以导出真实提现流水。

### 7. 内容审核仍是本地数据

**问题位置**

- `hanjing-clinic-管理后台源码-20260607/src/views/content/index.vue`
- `hanjing-clinic-管理后台源码-20260607/src/views/content/edit.vue`
- `hanjing-clinic-管理后台源码-20260607/src/views/content/banner.vue`

**当前表现**

- 科普文章/帖子由 `basePosts` 和 `Array.from` 生成。
- 审核通过、拒绝、置顶、删除只修改前端状态。
- 未调用内容审核接口。

**为什么不符合上线要求**

医疗健康内容和社区内容都涉及审核合规，必须服务端真实管理并可追溯。

**整改要求**

- 接入文章、帖子、评论、轮播图真实接口。
- 审核动作写入后端。
- 支持审核意见、上下架、置顶、删除、恢复。
- 医疗健康内容需要敏感词和资质审核。
- 轮播图上传需接对象存储，不能仅本地演示。

**建议验收标准**

- 小程序端只展示已审核通过内容。
- 下架或删除后小程序端实时不可见。
- 内容审核日志可追溯到操作人。

### 8. 医疗敏感数据缺少脱敏与数据权限控制

**问题位置**

- `hanjing-clinic-管理后台源码-20260607/src/views/patient/detail.vue`
- `hanjing-clinic-管理后台源码-20260607/src/views/appointment/emr.vue`
- `hanjing-clinic-管理后台源码-20260607/src/views/appointment/index.vue`
- `hanjing-clinic-管理后台源码-20260607/src/views/appointment/workbench.vue`
- `hanjing-clinic-管理后台源码-20260607/src/views/settings/log.vue`

**当前表现**

- 患者手机号、病史、过敏史、病历、诊断、订单等信息在页面中直接展示。
- 操作日志示例中包含“完整身份证号与就诊病历记录”文案。
- 未看到按角色控制字段可见性的前端逻辑。

**为什么不符合上线要求**

医疗健康数据属于高敏感数据。后台上线必须控制“谁能看、能看多少、看了是否留痕、能否导出”。

**整改要求**

- 手机号、身份证、银行卡、地址等默认脱敏。
- 只有授权角色可查看完整信息。
- 查看完整敏感信息需要二次确认或二次授权。
- 关键查看、导出、修改操作写入审计日志。
- 医生仅能访问与自己接诊相关患者。
- 门店账号只能访问本门店患者和订单。

**建议验收标准**

- 普通客服不能查看完整病历。
- 财务不能查看诊疗详情。
- 医生不能查看其他医生/门店无关患者。
- 敏感数据查看有审计记录。

### 9. 诊疗录入存在默认自动填充医疗结论风险

**问题位置**

- `hanjing-clinic-管理后台源码-20260607/src/views/appointment/emr.vue`

**当前表现**

- 页面会根据主诉自动预填诊断和医嘱。
- 模板中包含 OSAHS 诊断、AHI 数值、阻鼾器处方等医疗结论。
- 治疗建档、设备调节、随访任务可能在一次提交中串联创建。

**为什么不符合上线要求**

医疗结论和处方类内容必须由医生确认，不能让系统默认生成后造成误录。尤其是诊断、处方、治疗建议、设备前伸量等内容，需要明确医生责任和审核流程。

**整改要求**

- 自动填充内容应标识为模板建议，不能默认作为诊断结论。
- 提交前要求医生确认关键医疗字段。
- 诊断、处方、设备参数、随访计划分别校验。
- 设备前伸量等参数需设置合理范围和后端校验。
- 病历提交后修改应有版本记录。

**建议验收标准**

- 空诊断不能提交。
- 模板内容必须由医生主动选择或确认。
- 病历修改有版本留痕。

### 10. API 层过于基础，缺少生产级能力

**问题位置**

- `hanjing-clinic-管理后台源码-20260607/src/utils/request.ts`

**当前表现**

- 只有基础 baseURL、timeout、Authorization、错误提示。
- token 存储在 localStorage。
- 没有刷新 token、请求重试、取消请求、重复提交保护。
- 没有统一 requestId、链路追踪、错误上报。
- 没有模块化 API 文件，页面直接写接口路径。

**为什么不符合上线要求**

后台操作多、敏感度高，接口层需要更强的可靠性、可观测性和安全控制。

**整改要求**

- 增加 `/api/admin/me` 初始化校验。
- 建立模块化 API：auth、patient、appointment、order、distribution、content、settings。
- 对提交类接口做防重复点击和幂等控制。
- 对 401/403/500/网络异常做统一策略。
- 加入请求 ID、错误上报、接口耗时记录。
- 评估 token 存储策略，避免长期明文 localStorage 风险。

**建议验收标准**

- 页面不再直接散落接口路径。
- 重复点击不会产生重复订单、重复病历、重复审核。
- 后台错误能定位到接口、用户、时间和页面。

## 三、上线前建议整改的问题

### 1. 生产环境配置需要拆分

**问题位置**

- `hanjing-clinic-管理后台源码-20260607/.env`
- `hanjing-clinic-管理后台源码-20260607/.env.example`
- `hanjing-clinic-管理后台源码-20260607/vite.config.ts`

**当前表现**

- `.env` 和 `.env.example` 内容一致。
- `VITE_API_BASE_URL=http://localhost:5005`。
- 未看到 `.env.production`、`.env.staging`。

**整改建议**

- 建立 `.env.development`、`.env.staging`、`.env.production`。
- 生产环境使用 HTTPS API。
- `.env` 不应纳入版本管理。
- 构建时明确 mode。

### 2. 构建产物 chunk 过大

**检查结果**

执行 `npm run build` 成功，但 Vite 提示部分 chunk 超过 500KB：

- `index-0L9gJ_w2.js` 约 1126KB。
- `index-8OwAnh5F.js` 约 1428KB。

**影响**

首屏加载和弱网访问会变慢，尤其后台图表、TDesign、ECharts 等依赖容易进入大包。

**整改建议**

- 配置 `manualChunks` 拆分 Vue、TDesign、ECharts、业务页面。
- ECharts 按需引入。
- 路由组件已懒加载的基础上继续检查公共依赖打包。
- 开启 gzip/brotli 静态压缩。

### 3. 缺少前端错误监控和行为审计

**当前表现**

页面多处 `console.error`，但没有统一错误监控。

**整改建议**

- 接入前端错误监控。
- 记录接口错误、页面崩溃、资源加载失败。
- 关键操作如登录、导出、查看病历、修改病历、退款、提现、权限变更写入后端审计。

### 4. 缺少 403、404、500 页面

**当前表现**

路由中未看到无权限、页面不存在、系统错误页面。

**整改建议**

- 增加 403 无权限页。
- 增加 404 页面。
- 增加后端异常统一兜底页或空状态。
- 未授权路由不应静默进入空白或错误页面。

### 5. 表单校验不统一

**当前表现**

部分页面只做简单必填校验，部分页面没有明显校验。

**整改建议**

- 统一姓名、手机号、金额、日期、病历字段、设备参数校验规则。
- 重要操作增加确认弹窗。
- 服务端错误映射成可理解提示。
- 金额、佣金、退款、提现必须以后端计算结果为准。

### 6. 缺少导出与下载安全控制

**风险范围**

- 患者列表。
- 病历记录。
- 订单流水。
- 提现流水。
- 操作日志。

**整改建议**

- 导出需要权限控制。
- 导出敏感数据需要审批或二次确认。
- 导出任务写入审计日志。
- 导出文件设置有效期和访问鉴权。

### 7. 部署安全配置需要补齐

**整改建议**

- 全站 HTTPS。
- Nginx/网关配置 SPA fallback。
- 配置安全响应头：CSP、X-Frame-Options、X-Content-Type-Options、Referrer-Policy。
- 禁止暴露 sourcemap 或仅内部可访问。
- 静态资源设置缓存策略。
- API 与前端跨域策略最小化。

### 8. 依赖与仓库体积需要治理

**当前表现**

- 后台目录包含 `node_modules` 和 `dist`。
- 当前目录约 415MB。

**整改建议**

- 仓库不应提交 `node_modules` 和 `dist`。
- 依赖通过 lockfile 安装。
- 产物由 CI 构建生成。
- 上线前做依赖漏洞扫描。

## 四、开发环境配置上线前切换项

以下项目在开发阶段可以保留，但上线前必须切换并验证。

### 1. API 域名

**当前开发配置**

```env
VITE_API_BASE_URL=http://localhost:5005
```

**上线要求**

- 切换为生产 HTTPS API。
- 配置跨域白名单。
- 验证 token、cookie/header、代理路径均正常。

### 2. 登录和演示入口

**上线要求**

- 移除短信 mock 登录。
- 移除测试账号自动登录。
- 演示数据入口仅在开发环境可用。

### 3. 构建与发布

**上线要求**

- 使用 `npm ci` 安装依赖。
- 使用 `npm run build -- --mode production` 或明确生产构建脚本。
- CI 产出 dist。
- 静态资源部署到正式服务器或 CDN。

## 五、建议整改优先级

### P0：不处理不建议上线

- 移除短信 mock 登录。
- 路由守卫接入真实 `/api/admin/me`。
- 建立真实 RBAC 权限和数据范围控制。
- 管理员账号管理接真实接口。
- 权限管理接真实接口。
- 退款审核接真实接口和支付退款流程。
