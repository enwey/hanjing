# 微信小程序上线前问题清单

本文档基于当前 `mp-weixin` 目录进行检查，目标是对齐微信小程序官方开发规范、审核要求和生产运营要求。当前判断默认：本地 `127.0.0.1`、`urlCheck: false` 属于开发环境配置，不作为当前开发阶段问题；但上线前必须切换为正式配置并完成验证。

## 一、总体结论

当前小程序端已经具备微信小程序基本项目形态，包含 `app.json`、页面路由、分包、tabBar、自定义组件、API 层和状态管理，能用于开发调试和演示。

但距离“可提交审核、可生产运营”的标准仍有明显缺口，主要问题集中在：

- 生产环境仍深度依赖 mock 数据与 mock 兜底。
- 隐私协议、用户协议、隐私接口声明尚未完整闭环。
- AI 鼾声录音分析、社区、支付、订阅消息等核心业务能力未真实完成。
- 网络请求、登录态、错误处理、日志、安全缓存等生产能力还不够规范。
- 医疗健康类敏感数据处理不足，存在合规与安全风险。

## 二、上线前必须整改的问题

### 1. 生产包不能依赖 mock 数据

**问题位置**

- `mp-weixin/api/index.js`
- `mp-weixin/api/request.js`
- `mp-weixin/mock/index.js`
- `mp-weixin/mock/posts.js`

**当前表现**

- `api/index.js` 直接引入 `../mock/index.js`。
- 每个 API 方法都传入 `mockCallback`。
- `request.js` 在接口失败、401 重试失败、网络失败时会回退到 mock 数据。
- 社区页面直接读取 `mock/posts.js`。

**为什么不符合上线要求**

生产环境不能在接口失败时返回假数据，否则会造成：

- 用户看到虚假预约、订单、病历、评估、分销数据。
- 提交操作看似成功，但服务端没有真实记录。
- 后端故障被前端掩盖，运营和监控无法及时发现问题。
- 医疗健康场景下可能产生严重误导。

**整改要求**

- 生产构建彻底移除 mock fallback。
- mock 只允许在开发环境或独立 demo 模式启用。
- 接口失败时应返回明确错误状态、用户提示和日志上报。
- API 层应通过环境变量或构建常量区分 `development`、`trial`、`release`。

**建议验收标准**

- 断开后端服务时，页面显示真实失败提示，不出现 mock 数据。
- 生产包中不包含 `mock/index.js`、`mock/posts.js` 的业务引用。
- 所有核心接口都有真实服务端响应和错误处理。

### 2. 用户协议与隐私政策仍是占位内容

**问题位置**

- `mp-weixin/pages/auth/login.js`
- `mp-weixin/pages/auth/login.wxml`

**当前表现**

登录页的“用户协议”和“隐私政策”只是 `showModal` 弹出简短占位文案，例如“此处为鼾静健康诊所用户协议”。

**为什么不符合上线要求**

项目涉及手机号、地理位置、录音、健康评估、病历、就诊记录等个人敏感信息。微信审核和实际合规都要求：

- 用户可查看正式用户协议。
- 用户可查看正式隐私政策。
- 隐私政策内容应与实际收集、使用、存储、共享的信息一致。
- 微信后台“用户隐私保护指引”与小程序内声明保持一致。

**整改要求**

- 补充正式《用户协议》和《隐私政策》页面。
- 登录页协议入口跳转到正式页面，而不是占位弹窗。
- 隐私政策应明确说明手机号、定位、录音、健康信息、病历信息、订单信息等处理规则。
- 微信公众平台后台同步完善用户隐私保护指引。

**建议验收标准**

- 登录前可完整查看用户协议与隐私政策。
- 协议内容没有“此处为”“待补充”等占位文字。
- 小程序后台隐私接口声明与代码实际调用一致。

### 3. 隐私接口声明不完整

**问题位置**

- `mp-weixin/app.json`
- `mp-weixin/app.js`
- `mp-weixin/pages/assessment/recording/index.js`

**当前表现**

`app.json` 当前只声明：

```json
"requiredPrivateInfos": ["getLocation"]
```

但业务包含：

- 微信手机号授权登录。
- 地理位置门店选择。
- AI 鼾声录音分析。
- 健康评估与病历管理。

**为什么不符合上线要求**

微信会对隐私接口调用和后台隐私保护指引进行校验。涉及隐私接口或个人信息处理时，声明、授权、协议和实际功能必须一致。

**整改要求**

- 梳理所有涉及隐私的微信 API。
- 若上线真实录音，补齐录音相关授权、隐私声明和用户提示。
- 在调用隐私 API 前完成用户授权流程。
- 使用 `wx.getPrivacySetting`、`wx.openPrivacyContract`、`wx.onNeedPrivacyAuthorization` 等能力实现符合微信要求的隐私授权体验。

**建议验收标准**

- 微信开发者工具隐私检测无异常。
- 提交审核时不会因隐私接口与隐私协议不一致被拦截。
- 用户拒绝隐私授权后，有明确降级提示和功能限制。

### 4. AI 鼾声分析没有真实录音与上传

**问题位置**

- `mp-weixin/pages/assessment/recording/index.js`
- `mp-weixin/pages/assessment/recording/index.wxml`
- `mp-weixin/api/index.js`

**当前表现**

录音页当前主要是计时与动画：

- 点击麦克风只是启动计时器。
- 没有调用 `wx.getRecorderManager()`。
- 没有生成本地音频文件。
- 没有调用 `wx.uploadFile()` 上传音频。
- 提交时只传 `duration`。

**为什么不符合上线要求**

“AI 鼾声分析”是核心医疗健康卖点。如果没有真实录音文件和后端分析，当前功能只是演示，不具备生产能力。

**整改要求**

- 接入微信录音能力。
- 处理录音授权、开始、暂停、停止、失败、超时。
- 对录音时长、格式、文件大小进行限制。
- 上传音频文件到后端或对象存储。
- 服务端返回真实分析结果。
- 上传失败、分析失败、弱网中断需要可恢复。

**建议验收标准**

- 真机可完成录音、试听或重录。
- 服务端能收到真实音频文件。
- 分析结果来自真实后端任务，不是 mock。
- 用户拒绝麦克风权限时有清晰提示。

### 5. 社区功能仍是本地 mock

**问题位置**

- `mp-weixin/pages/community/index.js`
- `mp-weixin/pages/community/detail/index.js`
- `mp-weixin/pages/community/publish/index.js`
- `mp-weixin/mock/posts.js`

**当前表现**

- 帖子列表从 `mockPosts` 读取。
- 帖子详情从 `mockPosts` 和 `mockComments` 读取。
- 点赞、评论、发帖是前端内存操作。
- 页面刷新、换设备、重新进入后数据不可持久化。

**为什么不符合上线要求**

医患社区属于高风险内容场景，需要：

- 服务端真实数据。
- 内容审核。
- 敏感词过滤。
- 用户身份与医生身份校验。
- 举报、删除、风控能力。

**整改要求**

- 接入社区帖子、评论、点赞、发布接口。
- 加入内容审核状态：待审核、已通过、已驳回。
- 加入举报和删除机制。
- 区分普通用户、医生、专家身份。
- 敏感词、联系方式、医疗广告等内容需要审核策略。

**建议验收标准**

- 新发帖和评论刷新后仍存在。
- 未审核内容不会公开展示。
- 普通用户不能伪造医生/专家身份。

### 6. 支付链路缺失

**问题位置**

- `mp-weixin/pages/order/index.js`
- `mp-weixin/pages/order/detail.js`
- `mp-weixin/pages/product/index.js`
- `mp-weixin/pages/product/detail.js`

**当前表现**

代码中有订单、商品、支付状态、微信支付文案，但未看到 `wx.requestPayment` 调用。

**为什么不符合上线要求**

如果商城、预约、筛查服务或产品购买需要在线支付，必须完整接入微信支付闭环。

**整改要求**

- 后端创建支付订单。
- 前端调用 `wx.requestPayment`。
- 处理支付成功、失败、取消、超时。
- 后端接收支付回调并更新订单状态。
- 前端轮询或刷新订单状态。
- 退款、发货、售后状态需要与后端同步。

**建议验收标准**

- 真机能完成微信支付。
- 用户取消支付后订单状态正确。
- 支付成功以后服务端订单状态可靠更新。
- 重复点击不会产生重复支付单。

### 7. 订阅消息缺失

**问题范围**

- 预约成功提醒。
- 改约提醒。
- 就诊提醒。
- 订单发货提醒。
- 设备维护提醒。
- 复诊/随访提醒。

**当前表现**

未发现 `wx.requestSubscribeMessage` 调用。

**为什么不符合上线要求**

预约和医疗服务类小程序通常需要订阅消息承接关键履约通知。没有订阅消息会影响履约体验，也会增加客服成本。

**整改要求**

- 配置订阅消息模板。
- 在预约成功、改约、下单、维护申请等关键节点请求用户订阅。
- 后端保存订阅状态并发送消息。
- 对用户拒绝订阅做降级提示。

**建议验收标准**

- 用户完成预约后可订阅就诊提醒。
- 服务端能在预约前发送提醒。
- 拒绝订阅不会阻塞主流程。

### 8. 首页存在绕过统一请求层的网络请求

**问题位置**

- `mp-weixin/pages/index/index.js`

**当前表现**

首页统计直接调用 `wx.request`，没有走统一 `api/request.js`。

**为什么不符合上线要求**

绕过统一请求层会导致：

- 环境切换不一致。
- token 和鉴权策略不一致。
- 错误处理不一致。
- 日志和埋点缺失。
- 后续维护困难。

**整改要求**

- 新增 `getHomeStats` API 方法。
- 首页统一调用 API 层。
- 去除页面内硬编码 URL。

**建议验收标准**

- 全项目不再出现业务页面直接拼接 API 域名。
- 所有业务请求均经过统一请求封装。

### 9. 登录态与鉴权策略不够完整

**问题位置**

- `mp-weixin/stores/index.js`
- `mp-weixin/api/request.js`
- `mp-weixin/pages/auth/login.js`

**当前表现**

- 主要依靠本地 `access_token` 判断登录状态。
- 401 后尝试静默登录。
- 并发多个请求同时 401 时没有统一队列。
- refresh token 存在 mock 返回，但没有完整刷新流程。
- 登录失败后的跳转和重试策略不统一。

**为什么不符合上线要求**

生产环境中 token 过期、网络抖动、多接口并发很常见。鉴权处理不稳定会导致用户频繁掉线、重复登录或请求风暴。

**整改要求**

- 明确 access token 和 refresh token 生命周期。
- 401 时只允许一个刷新/重登流程，其余请求排队等待。
- 刷新失败统一清理登录态并跳转登录。
- 登录后回到原始目标页面。
- 不同权限页面统一做登录校验。

**建议验收标准**

- token 过期时，多个并发请求不会触发多次登录。
- 登录成功后回到原操作页面。
- 登录失败不会展示 mock 用户。

### 10. 医疗健康敏感数据保护不足

**问题范围**

- 手机号。
- 家庭成员信息。
- 病历记录。
- 睡眠评估结果。
- 鼾声音频。
- 订单和地址信息。

**当前表现**

从前端代码看，尚未形成完整的敏感数据保护策略。

**为什么不符合上线要求**

医疗健康类信息属于高敏感数据。上线前不仅要满足微信审核，还要考虑个人信息保护、医疗数据合规和运营安全。

**整改要求**

- 手机号、身份证、地址等敏感字段前端脱敏展示。
- 本地缓存最小化，不缓存不必要的病历和音频信息。
- 退出登录时清理敏感缓存。
- 禁止将敏感信息打印到 console。
- 上传音频时使用 HTTPS、鉴权、文件大小限制和后端安全校验。
- 后端接口按用户身份校验数据归属。

**建议验收标准**

- 普通用户无法访问他人病历、订单、评估记录。
- 本地缓存中没有不必要的明文敏感数据。
- 日志中没有手机号、病历、音频 URL 等敏感内容。

## 三、上线前建议整改的问题

### 1. 清理生产日志

**问题位置**

- `mp-weixin/app.js`
- `mp-weixin/pages/auth/login.js`
- `mp-weixin/stores/index.js`
- `mp-weixin/api/request.js`

**当前表现**

存在大量 `console.log`、`console.warn`、`console.error`。

**整改建议**

- 开发环境保留详细日志。
- 生产环境只保留必要错误上报。
- 授权事件、登录事件、用户信息、接口详情不要直接打印。
- 接入微信 RealtimeLogManager 或自建日志平台。

### 2. SourceMap 与代码保护策略待确认

**问题位置**

- `mp-weixin/project.config.json`

**当前表现**

`uploadWithSourceMap` 当前为 `true`。

**整改建议**

- 生产环境确认是否上传 SourceMap。
- 若上传，需要限制后台访问权限。
- 医疗类业务建议配合代码压缩、代码保护、敏感逻辑后端化。

### 3. 缺少 sitemap 配置

**问题范围**

- 当前未发现 `mp-weixin/sitemap.json`。

**影响**

不一定阻断上线，但会影响微信搜索索引和页面收录控制。

**整改建议**

- 添加 `sitemap.json`。
- 对公开页面允许索引。
- 对登录态、病历、订单、个人中心等页面禁止索引。

### 4. 自定义导航需要真机兼容验证

**问题位置**

- `mp-weixin/app.json`
- `mp-weixin/components/base/hj-navbar.js`

**当前表现**

全局启用 `navigationStyle: custom`。

**整改建议**

- 验证 iPhone 刘海屏、灵动岛、安全区。
- 验证 Android 不同状态栏高度。
- 验证微信胶囊按钮避让。
- 验证返回按钮、首页按钮、标题过长等状态。

### 5. 分包策略可以继续优化

**当前表现**

已有订单、分销、直播、社区分包，方向正确。当前包体约 2.8MB，短期不是阻断项。

**整改建议**

- 将低频页面继续放入分包。
- 静态 demo 图片和视频只保留生产需要的资源。
- 产品图、医生头像、直播封面建议由 CDN 下发。
- 避免生产包包含演示资源。

### 6. 页面异常态和空状态需要补齐

**问题范围**

- 网络失败。
- 无数据。
- 加载中。
- 未登录。
- 无权限。
- 提交失败。
- 重复提交。

**整改建议**

- 每个核心页面都要有 loading、empty、error、retry 状态。
- 表单提交按钮需要防重复点击。
- 弱网情况下给出明确反馈。

### 7. 业务表单校验需要统一

**问题范围**

- 预约。
- 家庭成员。
- 个人资料。
- 设备反馈。
- 提现。
- 社区发帖。

**整改建议**

- 统一必填校验。
- 手机号、金额、姓名、备注长度等格式校验。
- 服务端错误需要映射成用户可理解的提示。

### 8. 审核材料与资质需要提前准备

**业务风险**

项目属于医疗健康/诊所服务场景，微信审核可能关注：

- 主体资质。
- 医疗/健康服务说明。
- 商品售卖资质。
- 医生/专家身份展示。
- 诊疗、评估、AI 分析描述是否存在夸大宣传。
- 用户隐私保护指引是否完整。

**整改建议**

- 准备营业执照、医疗相关资质、门店信息。
- 明确 AI 鼾声分析的能力边界，避免诊断承诺。
- 医生/专家内容需要真实资质支撑。
- 商品详情避免医疗功效夸大。

## 四、开发环境配置上线前切换项

以下项目在开发阶段可以保留，但上线前必须切换并验证。

### 1. 网络域名与 HTTPS

**当前开发配置**

- `project.config.json` 中 `urlCheck: false`。
- `request.js` 中 `develop` 使用 `http://127.0.0.1:5005/api/v1`。

**上线要求**

- `release` 使用真实 HTTPS 域名。
- 域名配置到微信公众平台“开发设置-服务器域名”。
- 关闭“不校验合法域名、TLS 版本以及 HTTPS 证书”。
- 真机关闭调试模式验证网络请求。

### 2. 环境变量与构建开关

**上线要求**

- `develop`、`trial`、`release` 环境明确区分。
- mock 仅开发环境可用。
- 日志、调试工具、测试入口仅开发环境可用。
- 构建产物可追溯版本号。

### 3. AppID 与后台配置

**上线要求**

- 使用正式小程序 AppID。
- 配置服务器域名、业务域名、隐私保护指引。
- 配置订阅消息模板。
- 如涉及支付，完成微信支付商户号绑定。

## 五、建议整改优先级

### P0：不处理不建议提审

- 移除生产 mock fallback。
- 补正式用户协议和隐私政策。

- [ ] 正式隐私政策可访问。
